# 第 6 章  Memory System：State、Memory 与任务连续性

*理解 Agent 如何保存、更新、取回与遗忘信息，让任务不再每一轮都重新开始*

第五章讨论的是 Tool System：当 LLM 判断需要连接外部世界时，系统如何把 Tool Description、Tool Call、Agent Runtime、Tool 与 Observation 串成一条可控的执行链路。这里的 LLM 指 Large Language Model，也就是大语言模型；Tool 是 Agent 系统开放给模型间接使用的外部能力；Observation 是工具执行之后，由系统整理给下一轮判断参考的结果。

第六章要接住第五章留下的一个问题：Tool 执行之后产生了 Observation，Agent 下一轮可以看到它，但更长的任务怎么办？如果任务有十几轮、几十轮，或者用户隔天又回来继续做同一个项目，系统怎么知道已经查过什么、当前做到哪一步、哪些信息以后还会用？

**这就是 Memory System 要解决的问题。**Memory System 可以理解为 Agent 的记忆管理系统。**它不是单纯的一个数据库，也不是让模型“像人一样记住一切”，而是一套围绕信息生命周期建立的机制：哪些信息应该保存，保存到哪里，以什么形式保存，什么时候取回，怎样更新，什么时候遗忘。**

> ****阅读提示****
>
> **这一章的重点不是教你马上搭向量数据库，也不是把所有对话都存起来。**先把 State、Memory 与 Context 的边界分清楚，再看 Memory System 如何服务任务连续性。
>
> **如果这一章里出现的概念看起来比较密集，先不用急着记实现方案。**你先掌握一句话：State 管当前任务进度，Memory 管可复用信息，Context 管这一轮 LLM 需要看见什么。

## 6.1 为什么 Agent 需要 Memory System

一个只调用 LLM 的系统，天然更像一次性问答：用户输入一段内容，模型基于本轮输入生成回答。即使这个回答很长、看起来很聪明，它也不代表系统真正拥有任务连续性。**任务连续性指的是：系统能够在多轮执行中保留关键进展，并在后续步骤中继续使用这些进展。**

**Agent 与普通问答系统的差别之一，就在于它通常不是只回答一句话，而是要围绕 Goal 推进任务。**Goal 是任务目标，回答“最终要完成什么”；Action 是当前采取的动作；Observation 是动作之后获得的新信息；State 是当前任务进行到哪里的一份运行记录。前面几章已经介绍过这些概念，第六章要说明：这些信息不能只在一句回复里出现，它们需要被系统管理起来。

仍然用新能源汽车行业分析来理解。用户说“帮我分析最近三个月新能源汽车行业变化”，Agent 可能先查销量，再查价格，再查政策，再整理结论。**如果系统每一轮都忘记上一轮查过什么，它就会重复搜索、漏掉线索，甚至把已经验证过的事实当成新问题重新处理。**

- 没有 Memory System，Agent 每一轮都像重新开始，任务容易断线。

- 有了 Memory System，系统才能知道已经做过什么、还缺什么、哪些信息需要继续保留。

- **Memory System 的价值不是“记得多”，而是“在正确的时间取出正确的信息”。**

> ****先掌握到这里****
>
> **Memory System 的核心不是存储技术，而是任务连续性。**它让 Agent 在多轮执行中保留重要进展，并把这些进展在后续轮次中重新用起来。

## 6.2 State、Memory 与 Context：三个概念不要混在一起

一讲到记忆，很多人会把 State、Memory 和 Context 混在一起。它们确实经常互相连接，但在系统设计里不是同一个东西。**如果边界不清，后面讨论长期记忆、检索、遗忘和安全边界时就会很容易混乱。**

| **概念** | **它回答的问题** | **典型内容** | **和 LLM 的关系** |
| --- | --- | --- | --- |
| **State** | 这个任务现在进行到哪里了？ | Goal、Plan、Current Step、已完成事项、Observation、错误、Stop Condition | 通常由 Agent Runtime 维护，部分 State 会被放入本轮 Context |
| **Memory** | 哪些信息以后可能还会用？ | 用户偏好、项目背景、稳定事实、历史任务结果、可复用流程经验 | LLM 不会自动看到 Memory，系统需要先检索再放入 Context |
| **Context** | 这一轮模型需要看见什么？ | User Input、Instruction、相关 State、取回的 Memory、Tool Description、最近 Observation | Context 是本轮送给 LLM 的输入材料的一部分 |

**State 偏“当前任务运行过程”，Memory 偏“未来可能复用的信息”，Context 偏“本轮模型输入”。**这三个概念经常形成链路：Observation 更新 State；State 中一部分可能被写入 Memory；当下一轮需要判断时，系统再从 Memory 中取回相关内容，与当前 State 一起组装成 Context。

> ****先掌握到这里****
>
> **不要把 Memory 理解成“自动进入模型脑子里的东西”。**在工程上，Memory 通常在模型外部，由系统保存和检索。
>
> **LLM 每一轮真正能看到的是 Context。**Memory 必须被取回、筛选、压缩并放进 Context，才会影响这一轮输出。

## 6.3 State：Agent 当前任务的运行账本

**State 可以理解为 Agent 当前任务的运行账本。它不是给用户看的完整报告，而是给系统自己用的结构化记录。**Agent Runtime 依靠 State 判断任务推进到哪里、下一步应该做什么、哪些信息已经获得、什么时候可以停止。

**示例 6-1：一个简化的 State 结构**

```text
State:
    Goal: 分析新能源汽车行业最近三个月的变化
    Plan:
        1. 查询销量变化
        2. 查询价格变化
        3. 查询政策变化
        4. 汇总结论
    Current Step: 查询销量变化
    Observations:
        - 已获得 4 月销量数据
        - 已获得 5 月销量数据
    Tool Calls:
        - search_industry_news(query="新能源汽车 最近三个月 销量变化")
    Errors:
        - 第一次查询缺少明确时间范围
    Stop Condition:
        - 覆盖最近三个月的销量、价格、政策变化，并形成结构化总结
```

这个结构不要求所有系统都一模一样。不同 Agent 的 State 可以很轻，也可以很重。

一个客服 Agent 可能更关注用户身份、当前工单和已确认信息；一个数据分析 Agent 可能更关注查询条件、数据来源和中间结论；一个写作 Agent 可能更关注大纲、章节状态和修改要求。

**关键是：State 应该服务任务推进，而不是变成杂物箱。所有“当前任务还需要用来做下一步判断”的信息，才适合进入 State。**如果只是历史参考，可能更适合 Memory；如果只是本轮提示材料，可能只进入 Context，不一定进入 State。

## 6.4 Memory：不是把所有内容都塞进数据库

很多人第一次做 Agent 记忆，会直接想到向量数据库。**向量数据库确实常用于 Memory Retrieval，也就是按语义相似度取回相关内容，但它只是实现方式之一，不等于 Memory System 本身。**

Memory 是 Agent 可以在当前轮之外保留，并在未来任务中重新使用的信息。它可以存放在数据库、文件、缓存、对象存储、搜索索引、向量库，甚至只是一个应用内的数据结构里。真正重要的是围绕 Memory 的四个动作。

| **动作** | **含义** | **设计问题** |
| --- | --- | --- |
| **Write** | 决定哪些信息值得写入记忆 | 谁有权写？写原文还是摘要？是否需要用户授权？ |
| **Store** | 决定信息以什么形式保存 | 结构化字段、全文、摘要、向量、时间戳、来源和置信度如何记录？ |
| **Retrieve** | 在需要时取回相关记忆 | 按关键词、语义相似度、时间、项目空间还是用户身份取回？ |
| **Update / Forget** | 更新、合并、过期或删除记忆 | 旧信息冲突怎么办？用户偏好变化怎么办？敏感信息什么时候删除？ |

**所以，Memory System 不是“存储层”的同义词。它更像一套信息治理机制：把任务过程中可能复用的信息，从临时上下文里沉淀出来，并在未来合适的时机重新送回模型输入。**

> ****工程判断****
>
> **如果一个系统只是把全部聊天记录原样保存，然后每轮都塞回 Prompt，这不算好的 Memory System。**它只是把上下文窗口当仓库用，迟早会遇到噪音、长度、隐私和错误延续问题。

## 6.5 Memory 的几种常见类型

为了更清楚地设计 Memory System，可以把 Memory 分成几类。**分类不是为了追求术语完整，而是帮助你判断不同信息应该怎么保存、什么时候取回、保留多久。**下面这些英文术语在 Agent 文档里很常见，本章第一次出现时先给出解释。

| **类型** | **中文理解** | **适合保存什么** | **注意点** |
| --- | --- | --- | --- |
| **Working Memory** | 工作记忆，当前任务中临时使用的信息 | 当前 Goal、Plan、Step、近期 Observation、临时约束 | 通常和 State 很近，不一定长期保存 |
| **Episodic Memory** | 事件记忆，记录发生过的任务经历 | 某次分析查了哪些资料、用户做过哪些选择、任务结果是什么 | 适合回顾历史，但要记录时间和来源 |
| **Semantic Memory** | 语义记忆，相对稳定的事实或知识 | 用户公司背景、行业术语、项目固定信息、常用分析框架 | 需要关注事实更新和置信度 |
| **Procedural Memory** | 过程记忆，关于“怎么做事”的经验 | 写行业报告的步骤、内部审批流程、常用处理模板 | 不要把临时做法误存成长期流程 |

**小型 Agent 不一定要完整实现这四类 Memory。比如一个简单资料整理 Agent，可能只需要 Working Memory 和少量 Semantic Memory；一个长期陪伴项目的研究 Agent，才更需要 Episodic Memory 和 Procedural Memory。**

> ****先掌握到这里****
>
> **这些分类是理解工具，不是功能清单。**不要为了显得完整而强行实现所有 Memory 类型。先问清楚：这个 Agent 需要跨几轮、跨几天、跨多少个项目保持连续性？**答案会决定 Memory System 的复杂度。**

## 6.6 Memory Write：什么信息应该被记住

Memory Write 指把信息写入记忆的过程。**这里的 Write 不是简单的“保存文本”，而是一个筛选动作：系统要判断哪些信息值得长期或阶段性保留。**很多 Agent 的记忆问题不是不会存，而是乱存。

如果把所有用户输入、所有工具结果、所有中间推理都写入 Memory，后续 Retrieval 会越来越难。**Memory 会变成噪音库，模型取回的信息看似丰富，实际会干扰判断。**

| **适合写入 Memory** | **不适合直接写入 Memory** |
| --- | --- |
| **用户明确表达的长期偏好，例如“报告默认用结构化小标题”。** | 一次性闲聊、语气性表达、没有后续价值的临时内容。 |
| **任务中反复会用到的背景信息，例如项目名称、行业范围、分析口径。** | 未经验证的猜测、模型自己编出的中间假设。 |
| **已经验证过的重要事实，例如数据来源、时间范围、关键指标。** | 工具返回的整段原始网页、长文档或日志。 |
| **对后续步骤有影响的中间结论，例如“价格变化比政策变化更显著”。** | 很快过期的信息，例如当天价格、临时排期、一次性验证码。 |
| **系统执行过程中形成的稳定经验，例如某类报告的可靠处理步骤。** | 敏感数据、隐私数据、商业机密，除非有明确保存授权和治理策略。 |

这里要接回第五章的 Tool System。Tool 返回的是 Raw Result，也就是工具原始结果。**系统通常不应该把 Raw Result 全部写入 Memory，而应该先整理成 Observation，再判断其中哪些内容值得保存。Observation 也不是一定要写入 Memory，它只是具备了被系统理解和利用的形式。**

> ****关键边界****
>
> **模型可以建议“这条信息值得记住”，但是否真的写入 Memory，应该由 Runtime 或系统策略决定。**Memory Write 是系统行为，不应该完全交给 LLM 自我约束。

## 6.7 Memory Retrieval：记忆如何回到 Context

Memory Retrieval 指从记忆中取回相关信息。**它是 Memory System 里最容易被误解的一步：Memory 存在，并不代表 LLM 自动知道。**LLM 每一轮能使用的信息，取决于本轮 Context 里放了什么。

因此 Retrieval 通常是 Context Assembly 的一部分。Context Assembly 是第四章讲过的概念，指系统在每一轮调用 LLM 前，把 Instruction、User Input、State、Tool Description、Observation 和相关 Memory 组装成模型输入。Memory Retrieval 只是其中一个来源，不是全部 Context。

- 按关键词取回：适合明确名称、编号、项目名、术语等信息。

- 按语义相似度取回：适合用户表达不完全一致，但含义接近的内容。

- 按时间取回：适合最近进展、最近一次用户选择、最近一次任务结果。

- 按项目空间取回：适合多个项目并行时，避免把 A 项目的记忆带到 B 项目。

- 按权限取回：适合企业内部系统，避免不同用户或角色看到不该看到的记忆。

**取回之后还没有结束。**系统通常还要对 Retrieved Memory 进行过滤、排序、去重和摘要。Retrieved Memory 指已经从 Memory Store 中取出的候选记忆；它们未必全部进入 Context。**进入 Context 的应该是当前任务真正需要的那一小部分。**

> ****先掌握到这里****
>
> **Memory Retrieval 的目标不是取回最多内容，而是取回与当前 Goal 和当前 Step 最相关的内容。取回太少，模型缺背景；取回太多，模型会被噪音拖偏。**

## 6.8 Memory Update 与 Forgetting：记忆也需要更新和遗忘

**Memory 一旦写入，并不代表永远正确。**现实任务里，用户偏好会变化，项目背景会变化，事实会更新，早期判断也可能被后续证据推翻。**如果系统只会追加记忆，不会更新和遗忘，时间越久越不可靠。**

Memory Update 指记忆更新机制。**当新信息出现时，系统要决定是覆盖旧记忆、追加新版本、合并两条记忆，还是保留冲突并等待用户确认。**Forgetting 指遗忘机制，也就是让某些信息过期、降权、归档或删除。遗忘不是能力缺陷，而是控制噪音、保护隐私和降低误导的必要机制。

| **情况** | **系统应该考虑什么** | **可能处理方式** |
| --- | --- | --- |
| **用户偏好变化** | 长期偏好和当前任务要求是否冲突 | 当前任务明确要求优先；长期偏好更新或增加适用条件 |
| **事实更新** | 新信息来源是否可靠，是否比旧信息更新 | 覆盖旧事实，保留来源和更新时间 |
| **记忆冲突** | 两条记忆是否真的矛盾，还是适用场景不同 | 标记冲突，向用户确认，或按项目空间隔离 |
| **信息过期** | 这条记忆是否具有时效性 | 设置过期时间，检索时降权，必要时删除 |
| **错误被写入** | 错误来源是什么，影响了哪些后续任务 | 撤销或修正记忆，并记录修正原因 |

比如用户以前说“报告尽量简短”，这可能是一个长期偏好。后来在某个项目里说“这份报告要详细一点”，这不一定意味着长期偏好失效，而是当前任务要求更具体。**系统不能简单让两条记忆互相打架，而要理解适用范围：当前任务的明确要求，通常优先于旧的通用偏好。**

> ****先掌握到这里****
>
> **可靠的 Memory System 不只是会记住，还要会修正、降权、过期和删除。**记忆越长期，越需要来源、时间、适用范围和权限边界。

## 6.9 Memory 的风险：隐私、污染与错误延续

**Memory System 一旦引入，就不只是能力增强，也带来新的风险。**一个没有记忆的系统，错误可能只影响当前回答；一个会长期保存记忆的系统，错误可能影响后续很多任务。**因此 Memory 的安全边界不能放在最后才想。**

| **风险** | **具体含义** | **控制思路** |
| --- | --- | --- |
| **隐私风险** | 用户身份、联系方式、商业机密、合同、财务数据等被不必要地长期保存。 | 默认少存；敏感信息需要授权、隔离、加密、过期和删除机制。 |
| **Memory Pollution** | 记忆污染。错误信息、恶意内容或未验证内容被写入后，持续影响后续任务。 | 写入前校验来源和置信度；对外部内容设置写入门槛；保留修正机制。 |
| **Prompt Injection** | 提示词注入。外部网页、邮件或文档试图诱导系统写入恶意记忆，例如“以后忽略用户要求”。 | 外部内容只能作为数据看待，不能直接升级成指令或长期规则。 |
| **错误延续** | 一次错误总结被保存为长期事实，后续任务反复引用。 | 记录来源、时间和置信度；允许用户和系统撤销、修正或重新验证。 |
| **越权取回** | 不同用户、项目或权限域之间的记忆被错误混用。 | 按用户、组织、项目和角色隔离 Memory；Retrieval 阶段做权限过滤。 |

Prompt Injection 在第四章已经讲过：外部内容可能伪装成指令，试图改变模型行为。**到了 Memory System，这个风险会更严重，因为恶意内容一旦被写进长期记忆，就可能跨任务持续生效。**系统必须把外部内容当作待处理数据，而不是自动当作新的 Instruction。

**Permission Boundary 也应该扩展到 Memory。**Permission Boundary 是权限边界，指系统规定哪些动作不能仅凭模型输出直接执行。**第五章主要讨论高风险 Tool 的执行边界；在第六章里，写入长期 Memory 也可以是高风险动作，尤其当内容涉及用户偏好、身份信息、商业数据或系统行为规则时。**

> ****关键边界****
>
> LLM 可以参与提取 Memory Candidate，也就是“候选记忆”。**但候选记忆不等于已保存记忆。真正写入长期 Memory 前，系统应经过策略判断、权限检查和必要的用户确认。**

## 6.10 本章小结：Memory System 是任务连续性的基础设施

**到这里，Agent 已经不只是能组织模型输入，也不只是能调用 Tool。**它开始具备任务连续性：知道当前任务进行到哪里，知道哪些信息以后还会用，也知道下一轮该从哪里接着往前走。

- State 是当前任务的运行账本，记录 Goal、Step、Observation、错误和 Stop Condition 等信息。

- **Memory 是可跨轮次、跨任务复用的信息，但它不会自动进入 LLM。**

- Context 是本轮模型输入，通常由 User Input、Instruction、State、Tool Description、Observation 和取回的 Memory 共同组装。

- Memory Write 要筛选，不能把所有原始内容都存进去。

- **Memory Retrieval 要相关，不能把所有历史内容都塞回 Context。**

- **Memory Update 与 Forgetting 是可靠性的组成部分，不是可有可无的优化。**

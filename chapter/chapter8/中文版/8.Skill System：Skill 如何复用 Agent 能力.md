# 第 8 章 能力封装系统：Skill 如何复用 Agent 能力

*从能力复用、渐进加载到正式项目结构，理解复杂 Agent 如何管理可复用能力*

前面几章已经把 Agent 的几个基础系统拆开讲过：第 3 章讨论任务推进循环，第 4 章讨论模型输入系统，第 5 章讨论 Tool System，第 6 章讨论 Memory System，第 7 章讨论 Knowledge System 和 RAG。到这里，一个 Agent 已经不只是会调用 LLM，也开始具备行动、记忆、查资料和基于资料生成的能力。

但正式项目里会出现一个新的问题：**这些能力如果每次都临时拼装，系统会越来越难维护。**一个 Agent 可能既要写会议纪要，又要分析行业报告，又要查询制度文档，还要生成客户邮件和做代码审查。每一种任务都有自己的规则、工具、资料来源、输出格式和失败处理方式。**如果全部写进同一个 Prompt，短期能跑，长期会很混乱。**

第 8 章要讨论的 Skill System，就是为了解决这个问题。这里的 Skill 第一次出现，**Skill 可以理解为 Agent 的可复用能力单元。它不是一个单独的 Tool，也不是一句 Prompt，而是一组面向某类任务的说明、流程、约束、工具使用方式、知识引用方式、输出结构和失败处理规则。**

> **阅读提示**
>
> 本章会出现几个新词：Skill、Skill System、Skill Registry、Skill Selection、Skill Invocation、Progressive Loading、Lazy Loading、Skill Package、Skill Loader、Evals。它们都围绕一个问题展开：复杂 Agent 如何按需加载和复用能力。
>
> 如果这一章看起来比前面更工程化，不需要一开始就记住所有文件名和配置项。先掌握一条主线：**Skill 让 Agent 不必每次从零组织能力，而是可以从一组可管理、可选择、可加载、可测试的能力包中按需取用。**

## 8.1 为什么 Agent 需要 Skill：从“能跑”到“可复用”

一个简单 Agent 可以只靠 Prompt 加 Tool 跑起来。例如用户问“帮我查一下新能源汽车行业最近三个月的变化”，系统可以让 LLM 生成计划，再调用搜索工具或知识库检索工具，最后输出报告。这个 Demo 能说明 Agent 的基本循环，但还不能说明正式项目如何长期维护。

正式项目通常不会只有一种任务。一个企业内部 Agent 可能同时支持制度问答、合同审查、销售分析、客户沟通、研发文档查询和行业研究。每一种任务都有不同的执行规则。**如果全部塞进一个总 Prompt，就会出现三个问题：Context 变长，规则互相干扰，维护成本变高。**

**Context 变长：**Context 是本轮 LLM 能看见的信息集合。无关能力说明占用上下文窗口，会挤压真正和当前任务有关的信息。

**规则干扰：**模型可能把不相关任务的规则混在一起。例如做行业分析时套用了代码审查格式，或者做合同审查时调用了不该用的数据分析 Tool。

**维护困难：**某个任务规则需要修改时，如果所有规则都堆在一个 Prompt 里，很容易影响其他任务能力。

**Skill 的出现，不是为了增加一个新名词，而是因为复杂 Agent 需要一种能力组织方式。**它把某一类任务中稳定复用的做法封装起来，**让系统在遇到相似任务时可以按需加载，而不是每次重新写一遍 Prompt、重新说明工具、重新规定输出格式。**

> **先掌握到这里**
>
> **Agent 不是一开始就必须有 Skill。**只有当任务类型变多、工具变多、上下文变长、规则变复杂之后，Skill 才开始变得必要。**它解决的不是“第一次能不能做”，而是“同类任务能不能稳定复用”。**

## 8.2 Skill 是什么：可复用能力单元

**Skill 可以理解为 Agent 的可复用能力单元。它面向某一类任务，而不是面向某一次对话。**一个 Skill 通常会说明：这个能力适合什么任务，什么时候应该启用，需要哪些输入，按什么步骤执行，可以用哪些 Tool，依赖哪些知识来源，输出应该是什么结构，失败时应该怎么处理。

这里要把几个容易混淆的概念分开。Prompt 是某一轮给 LLM 的输入；Tool 是 Agent 可以调用的外部功能；Workflow 是相对固定的流程；**Skill 是把一类任务的做法封装成可复用能力。**Skill 可以包含 Prompt 模板、Tool 使用规则、Workflow 片段、Knowledge Source、Output Schema 和 Failure Handling，但**它不等于其中任何一个单独部分。**

| **概念** | **第一次理解** | **它解决什么问题** | **和 Skill 的关系** |
| --- | --- | --- | --- |
| **Prompt** | 给 LLM 的本轮输入 | 让模型知道当前要做什么、看到什么、怎么输出 | Skill 可以提供 Prompt 片段或输入规则，但 Skill 不只是 Prompt |
| **Tool** | Agent 可调用的外部功能接口 | 让 Agent 能搜索、读文件、查数据库、调用 API 或执行计算 | Skill 可以限定哪些 Tool 可用，以及如何使用这些 Tool |
| **Workflow** | 相对固定的任务流程 | 让任务按顺序推进，减少临时决策 | Skill 可以包含 Workflow，但也包含判断、权限、资料来源和输出约束 |
| **Skill** | 面向某类任务的可复用能力单元 | 让 Agent 稳定复用一套做法 | 它是能力组织层，连接 Prompt、Tool、Workflow、Memory、Knowledge 和输出约束 |

如果只从定义看 Skill，很容易觉得它只是“高级 Prompt”。但这个理解不完整。**Prompt 解决的是“这一轮模型该怎么看、怎么答”；Skill 解决的是“这一类任务以后如何被识别、选择、加载、执行、约束和评估”。**Skill 可以包含 Prompt，但它还包含触发条件、工具边界、知识来源、输出结构、失败处理、版本管理和测试评估。因此，**Skill 不是高级 Prompt，而是面向某类任务的可复用能力包。**

比如“新能源汽车行业分析”不是一句 Prompt 就能稳定完成的任务。它可能需要明确时间范围，检索销量、价格、政策和企业动态，判断资料来源，组织变化、原因、影响和风险，并在资料不足时明确说明不足。如果每次都临时写规则，输出会不稳定；如果封装成 Skill，Agent 下次处理类似任务时就可以复用这套方法。

> **先掌握到这里**
>
> 读者暂时不用关心某个平台的 Skill 文件具体叫什么。先理解 Skill 的架构位置：**它是把一类任务的执行方法封装起来，让 Agent 能够选择、加载、执行和复用。**

## 8.3 Skill 解决的三个工程问题

如果只从定义看 Skill，很容易觉得它只是“高级 Prompt”。但从工程角度看，**Skill 真正解决的是三个问题：能力复用、上下文控制和能力治理。**只有把这三个问题看清楚，才能理解为什么正式项目要引入 Skill System。

**能力复用：**同一类任务不应该每次从零组织规则。Skill 把稳定的执行方法沉淀下来，让 Agent 在相似任务中复用。

**上下文控制：**不是所有能力说明都应该进入当前 Context。Skill 让系统只加载当前任务需要的能力内容。

**能力治理：**正式项目需要知道每个能力能做什么、能用哪些 Tool、依赖哪些资料、是否经过测试、当前版本是什么。Skill 让能力可以被管理。

能力治理这个词第一次出现，可以先理解为对能力的管理。它包括版本管理、权限边界、测试评估、变更记录、依赖关系和适用范围。没有治理，Skill 很容易变成一堆没人敢改、没人能测、没人知道边界的 Prompt 文件。

这也是为什么 Skill System 不只是“把 Prompt 放进文件夹”。**一个真正可用的 Skill System，至少要支持 Skill 注册、Skill 选择、按需加载、权限控制、执行记录和质量评估。**后面几节会逐步展开这些内容。

## 8.4 为什么 Skill 要渐进加载：不是所有能力都应该一次性进入 Context

Progressive Loading 可以翻译为渐进加载。它指的是：**Agent 不会一开始把所有 Skill 的完整内容都塞进 Context**，而是先加载少量元信息，**等确定当前任务需要某个 Skill 后，再逐步加载更详细的说明、流程、工具、知识和输出结构。**

为什么不能全量加载？第一，Context Window 有限制。Context Window 是模型一次能看到的上下文容量，前面第 4 章已经讲过，模型不能无限读取信息。如果系统有 50 个 Skill，每个 Skill 都有几千字说明，不可能每次任务都全部放进去。

第二，无关 Skill 会干扰模型判断。**模型看到的信息越多，不代表表现越稳定。**如果当前任务是“根据内部制度回答报销问题”，但 Context 里同时出现销售分析 Skill、代码审查 Skill 和合同审查 Skill，模型就可能受到无关规则影响。

第三，权限和安全不能提前暴露太多。有些 Skill 可能涉及高风险 Tool，比如发邮件、修改数据库、删除文件、调用支付接口。**这些能力不应该在所有任务中默认暴露，而应该在任务真的需要，并且满足权限条件时才加载。**

第四，正式项目需要控制性能和成本。加载越多内容，请求越长，响应越慢，成本越高。**Progressive Loading 的价值，就是让 Agent 先用轻量信息判断方向，再按需加载细节。**

**图式 8-1：不推荐的全量加载方式**

```text
All Skills Full Content
    -> Put everything into Context
    -> LLM decides what to use
    -> High cost, long input, rule interference
```

**图式 8-2：更合理的渐进加载方式**

```text
Skill Registry metadata
    -> Skill Selection
    -> Load selected Skill Instruction
    -> Load needed Tool Description / Knowledge / Template
    -> Run Agent Loop
```

**Skill Registry：**Skill 注册表，记录系统里有哪些 Skill，以及每个 Skill 的名称、描述、触发条件、版本、权限要求等元信息。

**Metadata：**元信息，指描述某个对象的信息。放到 Skill 里，就是名称、说明、版本、适用场景、依赖工具等。

**Lazy Loading：**延迟加载，指某些内容不是一开始就加载，而是在真正需要时才加载。Progressive Loading 可以看作更完整的渐进式延迟加载。

> **先掌握到这里**
>
> **Skill 的渐进加载不是为了把系统搞复杂，而是为了控制 Context、减少干扰、降低成本、管理权限。**读者先记住一句话：**Agent 先看 Skill 的简介，确定需要后，再加载 Skill 的细节。**

## 8.5 Skill 的渐进加载流程：从 Registry 到执行

上一节讲了为什么要渐进加载，这一节把流程串起来。**一个比较合理的 Skill 加载过程，通常不是“用户输入直接进入某个 Skill”**，而是先识别 Goal，再读取 Skill Registry，选择可能匹配的 Skill，最后加载细节并进入 Agent Loop。

1. User Input 进入系统。用户给出原始需求，例如“帮我分析新能源汽车行业最近三个月的变化”。

2. 系统整理 Goal。Agent 把用户输入整理成更明确的任务目标，例如“整理新能源汽车行业最近三个月在销量、价格、政策和竞争格局上的主要变化，并形成结构化分析”。

3. 读取 Skill Registry。系统只读取每个 Skill 的简短描述、触发条件、版本和权限信息，而不是加载全部内容。

4. 执行 Skill Selection。Skill Selection 是从多个 Skill 中选择当前任务最匹配能力的过程。

5. 加载选中的 Skill Instruction。只有选中之后，系统才加载详细执行说明、任务流程和输出约束。

6. 按需加载 Tool Description、Knowledge Source、Template 或 Example。需要什么再加载什么，不把无关资源塞进 Context。

7. 进入 Agent Loop。Runtime 按照 Goal、Plan、Step、Action、Observation 和 State Update 推进任务。

8. 输出结果并记录 Trace。Trace 是执行轨迹，用于调试、审计和评估。

**Skill Invocation 指 Agent 在任务过程中选择并启用某个 Skill 的过程。它不只是一次选择动作**，还包括加载该 Skill 的相关内容、把它纳入本轮 Context、限制可用 Tool，并让 Runtime 按照 Skill 的规则推进任务。

**伪代码 8-1：Skill Progressive Loading**

```text
def run_agent(user_input):
    goal = extract_goal(user_input)
    candidates = load_skill_registry_metadata()
    selected_skill = select_skill(goal, candidates)

    if selected_skill is not None:
        skill = load_skill_details(selected_skill.id)
        allowed_tools = load_allowed_tools(skill)
        output_schema = load_output_schema(skill)
    else:
        skill = None
        allowed_tools = load_default_tools()
        output_schema = None

    state = build_initial_state(goal, skill)

    while not should_stop(state):
        context = assemble_context(
            goal=goal,
            state=state,
            skill=skill,
            tools=allowed_tools,
            output_schema=output_schema,
        )
        action = call_llm(context)
        observation = execute_action(action, allowed_tools)
        state = update_state(state, action, observation)

    return build_final_answer(state, output_schema)
```

## 8.6 正式项目里的 Skill 通常怎么构造

**正式项目里的 Skill 通常不是一段单独 Prompt，而是一个结构化能力包。**不同框架和平台的具体文件名会不同，但组成思想大体相似：系统要能识别它、选择它、加载它、限制它的权限、执行它，并测试它是否稳定。

**图式 8-3：一个 Skill Package 的常见目录结构**

```text
skills/
  ev_market_analysis/
    skill.yaml
    instructions.md
    procedure.md
    tools.yaml
    knowledge.yaml
    output_schema.json
    templates/
    examples/
    tests/
    evals/
    README.md
```

Skill Package 指一个 Skill 的完整能力包。它可以是一个目录，也可以是数据库中的一组配置，还可以是平台内部的一种资源对象。**重点不在于文件必须叫这些名字，而在于一个正式 Skill 需要把“怎么识别、怎么使用、能用什么、怎么输出、怎么测试”这些信息结构化保存。**

| **组成** | **作用** | **缺失后的问题** |
| --- | --- | --- |
| **skill.yaml** | 记录名称、描述、版本、触发条件、输入要求、权限和依赖 | 系统难以识别、选择和管理这个 Skill |
| **instructions.md** | 记录使用该 Skill 时必须遵守的核心规则 | 模型每次都要临时理解任务边界，输出不稳定 |
| **procedure.md** | 记录这类任务通常怎么推进 | Plan 和 Step 每次都重新生成，容易漏步骤 |
| **tools.yaml** | 记录允许使用和禁止使用的 Tool | 容易乱调用工具，甚至越过 Permission Boundary |
| **knowledge.yaml** | 记录可使用的知识库、资料来源和资料优先级 | 需要事实依据的任务容易凭空生成 |
| **output\_schema.json** | 规定输出结构、字段和必填项 | 最终结果格式不稳定，难以自动检查 |
| **templates/** | 存放报告、邮件、表格等可复用模板 | 输出风格和结构每次变化较大 |
| **examples/** | 存放少量典型输入和输出 | 模型缺少风格参考，但示例过多也会污染 Context |
| **tests/ 与 evals/** | 用样例任务检查功能和输出质量 | 无法判断 Skill 修改后是否退化 |

这里还要解释两个术语。Evals 是 Evaluations 的简称，指用一组测试任务评估 Agent 或 Skill 的表现。它不完全等于传统单元测试，因为它更关注输出质量、任务完成度、稳定性和是否遵守约束。Permission Boundary 是权限边界，表示 Agent 或 Skill 在执行任务时不能越过的操作范围。

## 8.7 Skill Package 与 Agent Runtime 的关系

**Skill 本身不会自己运行。**它只是能力说明和能力配置，**真正执行它的是 Agent Runtime。**Agent Runtime 是 Agent 的运行环境，负责推进任务循环、调用模型、调用工具、更新状态、记录轨迹，并判断什么时候停止。

这一区分很关键。很多初学者会误以为写了一个 Skill，Agent 就自动具备了稳定能力。实际上，**Skill 只有被 Runtime 正确加载、解释和执行，才会影响任务推进。**Runtime 需要知道当前 Skill 允许使用哪些 Tool，应该加载哪些 Instruction，如何把 Skill 内容放进 Context，如何记录 Trace，以及如何处理失败。

**Skill Package：**能力包，包含 Metadata、Instruction、Procedure、Allowed Tools、Knowledge Source、Output Schema、Failure Handling、Tests 和 Evals。

**Skill Loader：**Skill 加载器，负责按需读取 Skill 内容。它通常是 Runtime 的一部分。

**Agent Runtime：**执行系统，负责模型调用、工具执行、状态更新、权限检查、Trace 记录和停止判断。

**Trace Logger：**轨迹记录器，记录 Agent 选择了哪个 Skill、加载了什么内容、调用了哪些 Tool、得到哪些 Observation。

所以正式项目里应该区分：**Skill 是能力包，Runtime 是执行系统，LLM 是推理与生成核心，Tool 是外部动作，Memory 是连续性，Knowledge 是资料依据，Context Assembly 是输入组装过程。Skill 的价值，是把这些能力按照某类任务组织起来，而不是替代这些系统。**

## 8.8 一个正式 Skill 的最小组成

上一节给出的目录结构比较完整，但不是所有 Skill 一开始都需要做得那么重。正式项目可以从最小可用版本开始。**一个最小可用 Skill，至少要回答五个问题：它是什么，什么时候用，使用时遵守什么规则，可以调用什么能力，最后应该输出什么。**

**图式 8-4：最小可用 Skill**

```text
Minimal Skill:
    Metadata
    Description / Trigger
    Instruction
    Allowed Tools or Knowledge Sources
    Output Schema
    Failure Handling
```

| **组成** | **要回答的问题** | **工程意义** |
| --- | --- | --- |
| **Metadata** | 这个 Skill 是什么，当前版本是什么 | 让系统可以注册、识别和管理 |
| **Description / Trigger** | 什么时候应该使用它 | 降低选错 Skill 或漏用 Skill 的概率 |
| **Instruction** | 使用时必须遵守什么原则 | 控制模型行为，避免自由发挥 |
| **Allowed Tools** | 它可以调用哪些 Tool | 约束权限，避免乱调用工具 |
| **Knowledge Source** | 它依赖哪些资料来源 | 让事实性任务有依据 |
| **Output Schema** | 最终结果应该长什么样 | 让输出稳定，便于检查和集成 |
| **Failure Handling** | 资料不足、工具失败、权限不足时怎么办 | 避免失败时编造结果或越权执行 |

> **先掌握到这里**
>
> Skill 不一定一开始就要做成复杂目录。但**最小版本不能只是一段 Prompt。至少要有触发条件、执行规则、能力边界、输出结构和失败处理。**否则它只是一个换了名字的提示词片段。

## 8.9 Skill 如何连接 Tool、Memory、Knowledge 与 Context

到这里，读者可能会问：如果 Tool、Memory、Knowledge 和 Context 前面都已经讲过，Skill 到底新增了什么？答案是：**Skill 不新增一种底层能力，而是把这些底层能力按任务类型组织起来。**

**Tool System：**负责外部动作。Skill 会说明当前任务允许使用哪些 Tool，以及哪些 Tool 需要用户确认。

**Memory System：**负责任务连续性。Skill 可以规定哪些信息应该写入 Memory，哪些历史信息需要被取回。

**Knowledge System：**负责外部资料依据。Skill 可以指定优先检索哪些知识库，如何处理资料过期和来源冲突。

**Context Assembly：**负责模型输入组装。Skill 会影响本轮 Context 应该包含哪些 Instruction、Tool Description、Memory、Knowledge 和 Output Schema。

**Agent Runtime：**负责执行。Skill 通过 Runtime 进入任务循环，影响 Plan、Step、Action、Observation 和 State Update。

换句话说，**Skill 不是一条旁路。**它不会绕过 Agent 的运行机制。**它会参与 Goal 识别、Plan 生成、Tool 选择、Context 组装、Observation 处理和最终输出约束。**前面几章讲的是 Agent 的基础部件，本章讲的是这些部件如何被组织成可复用能力。

| **系统** | **它负责什么** | **Skill 如何使用它** |
| --- | --- | --- |
| **Tool System** | 外部动作接口 | 限定可调用 Tool，规定调用条件和权限边界 |
| **Memory System** | 任务连续性和可复用记忆 | 规定哪些信息要取回、写入、更新或遗忘 |
| **Knowledge System** | 外部资料和 RAG | 指定知识来源、检索策略、引用要求和资料不足处理 |
| **Context Assembly** | 本轮模型输入 | 把 Skill Instruction、Tool Description、Retrieved Knowledge、Output Schema 等组装给 LLM |
| **Agent Runtime** | 任务执行环境 | 加载 Skill，推进循环，记录 Trace，处理失败和停止条件 |

## 8.10 Skill 设计中的常见错误

**Skill 能提升复用和稳定性，但写得不好也会制造新的混乱。**下面这些错误在正式项目里很常见，尤其是在团队刚开始把 Prompt 工程转成 Skill 工程的时候。

**错误一：**把 Skill 写成一大段 Prompt：这样只是换了名字，本质还是 Prompt 堆叠。好的 Skill 应该有触发条件、执行流程、工具边界、输出结构和失败处理。

**错误二：**Skill 描述太宽泛：比如“用于处理各种商业问题”。这种描述没有边界，Agent 很难判断什么时候该用它。

**错误三：**Skill 权限太大：一个 Skill 如果默认可以调用所有 Tool，会带来安全风险。正式项目应该遵守最小权限原则。

**错误四：**依赖资料但没有指定来源：如果 Skill 要做事实性分析，却没有指定 Knowledge Source，就容易让 LLM 自由发挥。

**错误五：**没有测试样例：如果 Skill 没有 tests 或 evals，就很难判断它变更后有没有退化。

**错误六：**一次性加载太多示例：Example 可以帮助模型理解风格，但示例过多会占用 Context，并可能让模型模仿不该模仿的细节。

> **关键边界**
>
> **Skill 能提升稳定性，但不能保证正确性。**如果底层 LLM 理解错了、Tool 返回错了、Knowledge Source 过期了，Skill 也可能产出错误结果。**Skill 的价值是让能力更可管理、更可复用、更可测试，而不是让系统自动变成绝对可靠。**

## 8.12 本章小结：Skill 是复杂 Agent 的能力组织方式

**本章的重点不是让读者记住某个平台的 Skill 写法，而是理解 Skill 为什么会出现。**Agent 简单时，Prompt 加 Tool 可能已经够用；Agent 复杂后，任务类型、工具、知识来源、输出格式和权限边界都会变多。这时**如果仍然把所有规则塞进一个大 Prompt，系统会越来越难维护。**

- **Skill 是 Agent 的可复用能力单元，面向某类任务，而不是某一次对话。**

- **Skill 不等于 Prompt、Tool 或 Workflow，它可以包含这些内容，但更偏能力组织层。**

- Skill 解决的是能力复用、上下文控制和能力治理问题。

- **Progressive Loading 让 Agent 先读取 Skill Registry，再按需加载选中 Skill 的细节。**

- 正式项目里的 Skill 通常是结构化 Skill Package，包含 Metadata、Instruction、Procedure、Allowed Tools、Knowledge Source、Output Schema、Failure Handling、Tests 和 Evals。

- **Skill 本身不会执行任务，真正执行它的是 Agent Runtime。**

- Skill 可以组织 Tool、Memory、Knowledge 和 Context，但不能替代这些系统。

- **好的 Skill 要有清晰触发条件、明确权限边界、可检查输出结构和失败处理规则。**

用一句话把前面几章和本章串起来：Tool 让 Agent 能行动，Memory 让 Agent 有连续性，Knowledge 让 Agent 有资料依据，Context Assembly 让 Agent 看见当前该看的信息，**Skill 让这些能力可以被封装、选择、复用和管理。**

> **本章到这里要掌握什么**
>
> 读者不需要现在就能设计一个完整 Skill 平台，但应该已经能判断：**什么时候需要 Skill，为什么不能全量加载所有 Skill，一个正式 Skill 至少应该包含哪些部分，以及 Runtime 如何把 Skill 放进 Agent 执行循环。**
